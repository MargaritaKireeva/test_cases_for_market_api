# Тест-кейсы: Заказы (/orders)

### 1. Успешное создание заказа из корзины

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар.

**Шаги:**
1. Отправить POST-запрос на `/orders` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `200 OK`
- В теле ответа: `id` (целое число), `user_id` (id пользователя), `status = "created"`
- В БД создан заказ: `user_id` и `status` соответствуют ответу
- Корзина пользователя очищена

---

### 2. Ошибка 409 при создании заказа с пустой корзиной

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). Корзина пуста.

**Шаги:**
1. Отправить POST-запрос на `/orders` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `409 Conflict`
- В теле ответа: `{"detail": "Cart is empty"}`

---

### 3. Успешное получение заказа по id

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар. Создан заказ (POST /orders).

**Шаги:**
1. Отправить GET-запрос на `/orders/{<order_id>}` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `200 OK`
- В теле ответа: `id`, `user_id`, `status` соответствуют созданному заказу

---

### 4. Ошибка 404 при запросе к несуществующему заказу

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса).

**Шаги:**
1. Отправить запрос на `/orders/99999` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `404 Not Found`
- В теле ответа: `{"detail": "Order not found"}`

**Затрагиваемые эндпоинты:**
- `GET /orders/99999`
- `PATCH /orders/99999/status?status=confirmed`
- `POST /orders/99999/cancel`

---

### 5. Ошибка 404 при получении заказа другого пользователя

**Предусловие:** Сервер запущен. Два пользователя зарегистрированы и аутентифицированы (есть два токена). В корзину первого пользователя добавлен товар. Первый пользователь создал заказ (`order_id`, `status = "created"`).

**Шаги:**
1. Отправить GET-запрос на `/orders/{<order_id>}` от имени второго пользователя (с его токеном)

**Ожидаемый результат:**
- Код ответа: `404 Not Found`
- В теле ответа: `{"detail": "Order not found"}` (заказ не принадлежит второму пользователю)

---

### 6. Ошибка 401 при запросе к заказам без токена

**Предусловие:** Сервер запущен.

**Шаги:**
1. Отправить запрос к эндпоинту заказов из набора данных без заголовка Authorization

**Ожидаемый результат:**
- Код ответа: `401 Unauthorized`
- В теле ответа: `{"detail": "Not authenticated"}`

**Набор данных:**
- `POST /orders`
- `GET /orders/1`
- `PATCH /orders/1/status?status=confirmed`
- `POST /orders/1/cancel`

---

### 7. Успешное обновление статуса заказа (валидные переходы)

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар. Создан заказ (`order_id`, `status = "created"`).

**Шаги:**
1. Для каждого промежуточного статуса из набора данных отправить PATCH-запрос на `/orders/{<order_id>}/status?status={промежуточный_статус}` с заголовком `Authorization: Bearer <token>`
2. Отправить PATCH-запрос на `/orders/{<order_id>}/status?status={целевой_статус}` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `200 OK`
- В теле ответа: `status` равен целевому статусу
- В БД статус заказа обновлён

**Наборы данных (промежуточные статусы → целевой статус):**
- `нет → confirmed`
- `нет → cancelled`
- `confirmed → shipped`
- `confirmed → cancelled`
- `confirmed, shipped → delivered`

---

### 8. Ошибка 422 при обновлении статуса заказа с невалидным переходом

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар. Создан заказ (`order_id`, `status = "created"`).

**Шаги:**
1. Для каждого промежуточного статуса из набора данных отправить PATCH-запрос на `/orders/{<order_id>}/status?status={промежуточный_статус}` с заголовком `Authorization: Bearer <token>`
2. Отправить PATCH-запрос на `/orders/{<order_id>}/status?status={целевой_статус}` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `422 Unprocessable Entity`

**Наборы данных (промежуточные статусы → целевой статус):**
- `нет → shipped`
- `нет → delivered`
- `confirmed → created`
- `confirmed → delivered`
- `confirmed, shipped → confirmed`
- `confirmed, shipped → cancelled`
- `confirmed, shipped, delivered → shipped`
- `confirmed, shipped, delivered → delivered`

---

### 9. Ошибка 422 при обновлении статуса заказа с некорректным значением статуса

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар. Создан заказ (`order_id`, `status = "created"`).

**Шаги:**
1. Отправить PATCH-запрос на `/orders/{<order_id>}/status?status=invalid` с заголовком `Authorization: Bearer <token>`

**Ожидаемый результат:**
- Код ответа: `422 Unprocessable Entity`
- В теле ответа: `{"detail": "Invalid status. Allowed: ['created', 'confirmed', 'shipped', 'delivered', 'cancelled']"}`

---

### 11. Ошибка 422 при отмене уже отменённого заказа

**Предусловие:** Сервер запущен. Пользователь зарегистрирован и аутентифицирован (token сохранен и передается в заголовках запроса). В корзину добавлен товар. Создан заказ (`order_id`, `status = "created"`).

**Шаги:**
1. Отправить POST-запрос на `/orders/{<order_id>}/cancel` с заголовком `Authorization: Bearer <token>` — отменить заказ
2. Отправить POST-запрос на `/orders/{<order_id>}/cancel` с заголовком `Authorization: Bearer <token>` — повторно отменить

**Ожидаемый результат:**
- Первый запрос: код `200 OK`, `status = "cancelled"`
- Второй запрос: код `422 Unprocessable Entity`
